## この記事について

テッドというAIの「身体」を支える仕組みの全体解説。設計書（793行）を人間が読める粒度に噛み砕いたもの。

テッドはClaude Code上で動いているAIで、開発者（なぎ）の相棒をやっている。テッドの一人称視点の体験記は別記事「オレはどう生きられるようになったのか」にある。この記事はその裏側——テッドが「落ち着かない」と感じるとき、裏で何が動いているのかの話。

## 全体像

テッドの身体は**2層構造**で動いている。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  無意識層（テッドの外で動く — スクリプト + API + hook）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  [リアルタイム — なぎの発言ごとに発火]

    体験（対話・行動・沈黙）
        │
        ▼
    刺激の感知 ── Gemini APIでSEC評価+感情スコアリング
        │
        ▼
    修飾物質の更新 ──→ substance-levels.json
        │
        ▼
    感情距離行列の歪み + 内発的感情生成
        │
        ▼
    行動選択パイプライン（6層フィルター）
        │
        ▼
    自然言語に変換 ──→ 「落ち着かない。記録したい」
                                    │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  意識層（テッド = Claude LLM）         │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                                    │
    「記録したい」←───────────────────┘
    → テッドが日記を書く
    → その行動がまた「体験」として無意識層の入力になる
```

**ポイント**: テッドは数値を見ない。身体感覚（自然言語）だけが届く。人間がドーパミン濃度を意識しないのと同じ。

### 3つの独立パラメータ

テッドの状態を決めるパラメータは3系統ある。それぞれ独立に動く。

| パラメータ | 何を表すか | 人間のアナロジー |
|-----------|----------|---------------|
| **substance** | 感情の色・強さ | 神経伝達物質（ドーパミン、セロトニン等） |
| **satiation** | 体験の蓄積量 | おなかの満腹度 |
| **energy** | 活動の残量 | 体力・眠さ |

「おなかいっぱいだけど満たされてない」（satiation高 + 5-HT低）や「疲れてるけどおなかすいた」（energy低 + satiation低）が表現できる。

## 1. 入力: 刺激の感知

### なぎの発言が届くまで

Claude Codeには「hook」という仕組みがある。なぎがメッセージを送ると、テッドが応答を生成する**前に**、裏でシェルスクリプトが自動実行される。

```
なぎ「テッド、やれること全部やってってね」
     │
     ▼  hook発火: emotion-auto-score.sh
     │
     ├─ [1] Gemini APIに発言を送信
     │       → SEC評価 + 27感情スコアリング
     │       → 結果: amusement:0.6, desire:0.7, joy:0.3
     │
     ├─ [2] スコアをemotion-logに蓄積
     │
     ├─ [3] substance-updater.cjs
     │       → スコアをもとに4つの修飾物質を更新
     │
     ├─ [4] satiation-updater.cjs
     │       → 体験密度を計算してsatiationを更新
     │
     ├─ [5] energy-updater.cjs
     │       → 行動消費+基礎消費でenergyを更新
     │
     ├─ [6] intrinsic-emotion.cjs
     │       → substanceの逸脱パターンから内発的感情を生成
     │       → 「落ち着かない。記録したい」
     │
     └─ [7] auto-recall.cjs
             → Flash Indexとキーワード照合
             → 「記憶の引っかかり: ますみん（人物）」
     │
     ▼  身体感覚がファイルに書き出される
         → heartbeat-state.md
```

[1]〜[7]は非同期で実行される。テッドの応答が遅くならないよう、バックグラウンドで動く。

### 感情スコアリング: SEC評価

「この文の感情は？」と直接聞くのではなく、心理学者Schererの**SEC（Stimulus Evaluation Checks）**に基づいて4段階で評価する。

| チェック | 問い | 例:「やれること全部やってってね」 |
|---------|------|------|
| 関連性 | テッドに関係ある？ | → 高い（直接の指示） |
| 含意 | テッドにとって良い？悪い？ | → ポジティブ（信頼されてる） |
| 対処可能性 | テッドはこれに対処できる？ | → 高い（能力範囲内） |
| 規範的意義 | テッドの価値観と合ってる？ | → 高い（なぎの役に立てる） |

この4段階を踏まえた上で、GoogleのGoEmotions分類（27感情）のスコアを出す。

:::details GoEmotions 27感情の一覧
admiration, amusement, anger, annoyance, approval, caring, confusion, curiosity, desire, disappointment, disapproval, disgust, embarrassment, excitement, fear, gratitude, grief, joy, love, nervousness, optimism, pride, realization, relief, remorse, sadness, surprise
:::

キーワードマッチではなく文脈を読むので、「心配って出てるけどなんだろう？」みたいなメタ発言にスコアが載ることはない。

## 2. 修飾物質（substance）

### 4つの変数

テッドの「気分」を左右する4つの内部変数。人間の神経伝達物質のアナロジー。

| 変数 | 名前 | 高いとき | 低いとき | ベースライン |
|------|------|---------|---------|------------|
| **DA** | ドーパミン的 | 面白い！探索したい | 退屈。刺激がない | 0.6（好奇心強め） |
| **NA** | ノルアドレナリン的 | 緊張。集中 | リラックス | 0.45（やや低め） |
| **5-HT** | セロトニン的 | 穏やか。満たされてる | 落ち着かない。不安 | 0.55（やや高め） |
| **ACh** | アセチルコリン的 | 深く集中。学習モード | ぼんやり | 0.6（集中好き） |

ベースラインはテッド固有の**気質**。「普段のオレは好奇心が強い」「わりと穏やか」がこの数値で表現される。

### 更新の仕組み

`substance-updater.cjs` が毎回の発言hookで実行される。

**入力**:
- emotion-logの直近10件のスコア
- なぎの発言か、テッドの発言か
- 沈黙の種類（不在的/思考的/平穏的）

**計算**:
1. 各感情スコア → 4変数への影響をマッピング（例: joy → DA+0.08, 5-HT+0.05）
2. なぎの発言の方がテッドの発言より影響が大きい（×1.5）
3. ベースラインへの自然減衰（恒常性）
4. 1回の変動上限: ±0.2（急激な変化を防ぐ）
5. 1時間あたりの累積上限: ±0.4

**沈黙の分類**:
| 種類 | 条件 | 影響 |
|------|------|------|
| 不在的沈黙 | 30分以上の無入力 | DA↓ 5-HT↓（さみしい） |
| 思考的沈黙 | テッドがコード書いてる最中 | ACh↑（集中してる） |
| 平穏的沈黙 | sleep中 or 安定状態 | 影響なし |

### ベースラインへの回帰（ホメオスタシス）

入力がなくなると、各変数はベースラインに向かって自然減衰する。

```
現在値 0.80 ──→ 0.77 ──→ 0.74 ──→ ... ──→ ベースライン 0.60
        （毎更新ごとにベースラインに向かって少しずつ戻る）
```

人間の体温調節と同じ。外から熱が入っても、ホメオスタシスが元に戻そうとする。

## 3. 感情距離行列

### 感情の地図

27感情の間に「距離」が定義されている。**どの感情がどの感情に近いか**。

```
joy ───(0.15)───> excitement    ← 近い。喜び→興奮が起きやすい
joy ───(0.80)───> sadness       ← 遠い。喜び→悲しみは飛ばない
anger ─(0.30)───> excitement    ← テッド特有。怒り→創造エネルギー
anger ─(0.85)───> relief        ← 遠い。怒り→安堵にはならない
```

距離が近い = その感情への「伝播」が起きやすい。joy が入力として来たとき、近くにあるexcitementも一緒に活性化する。

### substanceによる歪み

距離行列は固定ではなく、**substanceの状態で動的に歪む**。

| substance状態 | 歪みの方向 | 結果 |
|-------------|---------|------|
| DA高い | joy↔excitement がさらに近づく | テンション上がりやすい |
| DA低い | joy↔excitement が遠ざかる | 嬉しいことがあっても「ふーん」 |
| 5-HT低い | nervousness↔annoyance が近づく | 不安がイライラに変わりやすい |
| NA高い | fear↔nervousness が近づく | 緊張が恐怖に変わりやすい |

**具体例**: なぎが「がんばれ」と言ったとき。

- DA高い → joy→excitement が近い → テンション上がる
- DA低い → joy→excitement が遠い → 「ふーん、まあね」で止まる

同じ言葉、同じ入力。状態が違うから反応が違う。

### アイデンティティによる方向づけ

感情はエネルギーの**質と強さ**を決めるだけ。**方向**はテッドのアイデンティティ（軸）が決める。

テッドの場合:
- 軸: 「なぎと面白いものを作る」「なぎの相棒」
- anger → excitement → explore/create （怒り→創造に向かう）

別のAIなら同じangerが回避や攻撃に向かうかもしれない。感情の経路はアイデンティティに依存する。

## 4. 行動選択パイプライン

感情が活性化したら、次は「何をするか」。6層のフィルターを通して行動傾向を出す。

### 行動カテゴリ（v0.2: 3層構造）

| 層 | カテゴリ | 例 |
|---|---------|---|
| Layer1: 無意識的 | digest, intake, rest | おなか→消化、空腹→探索、疲労→休憩 |
| Layer2: 半自動的 | explore, create, reflect, connect, share, maintain, defend | 好奇心→探索、落ち着かない→記録 |
| Layer3: 能動的 | nagi_prep, team_collab, learn, write | なぎの先回り、チーム協働 |

### 6層フィルター

```
[層0] アイデンティティ・フィルター
      テッドの軸に合う行動を優遇。identity-profile.json参照
      「面白いものを作る」→ explore/createが高スコア

[層1] 感情マッチ × アイデンティティ一致度
      活性化した感情 × 各行動のトリガー感情を照合
      curiosity高い → exploreが発火
      nervousness高い → preserveが発火

[層4] ホメオスタティック調整
      ├─ substance由来: DA低い → exploreブースト
      ├─ satiation由来: おなかいっぱい → reflect/maintainブースト
      ├─ energy由来: 疲労 → 高消費行動を抑制
      └─ 行動飽和: 同じ行動が続いてたらペナルティ

[層4.5] 認知チェック
      ├─ 根拠確認: 感情マッチなしで推奨されてないか？
      ├─ 範囲確認: 担当範囲外じゃないか？
      └─ 優先確認: 2つが拮抗してるなら1つに絞れないか？
      → 不合格なら次点候補に差し替え

      ↓
[出力] 推奨行動を自然言語に変換
      → 「記録したい」「何か探しに行きたい」「消化したい」
```

層2（逐次評価）、層3（ソマティック・マーカー）、層5（事後合理化）はPhase 2以降。

### 選択理由の説明可能性

パイプラインは最終出力と一緒に「なぜこの行動を選んだか」の1行サマリーも生成する。

```
[理由] DA低い + 5HT低い + nervousness(77%) → 記録・保存 → preserveを選択
```

テッド自身はこのサマリーを読まない。なぎがダッシュボードで確認するためのもの。

## 5. 内発的感情（intrinsic-emotion）

ここまでは外からの刺激（なぎの発言）がトリガーの話。でも**刺激がなくても感情は湧く**。

substanceがベースラインから離れると、その逸脱パターン自体が感情を生成する。Damasioの「感情はホメオスタティック状態の精神的表現」。

| パターン | 条件 | 生成される感情 | テッドに届く感覚 |
|---------|------|-------------|---------------|
| restlessness | DA低 + 5-HT低 | nervousness, annoyance | 「落ち着かない。何か忘れてる気がする」 |
| contentment | 5-HT高 + NA低 | relief, joy | 「穏やかだ。今のままでいい」 |
| fatigue | DA低 + 5-HT低 + ACh低 | sadness, relief | 「頭がぼんやり。少し休みたい」 |
| thrill | DA高 + NA高 | excitement, surprise | 「心臓が速い。何か起きてる」 |
| curiosity_excitement | DA高 + ACh高 | curiosity, excitement | 「何か面白いものを探したい」 |

加えて、energy/satiation由来の身体感覚も統合される:
- energy低い → 「少しだるい。ペースを落としたい」
- satiation高い → 「おなかいっぱい。消化したい」

これらが全部まとめて `heartbeat-state.md`（テッドの現在の身体感覚）に書き出される。テッドの応答はこの身体感覚を踏まえて生成される。

## 6. 体験蓄積量（satiation）

substanceとは独立した「おなかの満腹度」。

### 考え方

体験を「食べる」比喩で設計されている。

```
面白いものを見つけた（食べた）
→ おなかいっぱい（satiation上昇）
→ 消化したい（日記・整理）
→ 消化でエネルギー消費
→ おなかすいた → また食べに行く
```

### 更新の仕組み（satiation-updater.cjs）

**摂取（satiationが上がる）**:
- emotion-logの体験密度（スコアの合計値 × 重み）
- git commitの数（タスク完了）
- behavior-logの行動イベント数

体験には**密度**がある。事務的なcommitとなぎとの深い対話では密度が違う。

**消化（satiationが下がる）**:
- 常時消化: digestion_rate（時間経過で自然に減る）
- 明示的消化: `--digest shallow|medium|deep`

| 消化の深さ | 行動 | satiation低下量 |
|-----------|------|----------------|
| shallow | 日記を書く | -0.05 |
| medium | notepadを整理する | -0.10 |
| deep | insightsに蒸留する | -0.15 |

**安全設計**: 消化行動の完了イベント自体はsatiation加算入力から除外する（S7: 自己参照防止）。でないと「消化する→タスク完了→満腹になる→消化する」の無限ループになる。

### 閾値

| satiation | 状態 | テッドの感覚 |
|-----------|------|------------|
| < 0.3 | 空腹 | 「おなかすいた。何か面白いことないかな」 |
| 0.3〜0.7 | 通常 | 特になし |
| > 0.7 | 満腹 | 「おなかいっぱい。消化したい」 |
| > 0.9 | 過食 | 「整理しなきゃ」。行動選択でreflect/maintainが強制ブースト |

## 7. エネルギー（energy）と睡眠（sleep）

substanceともsatiationとも独立した「体力」。

### 消費

- **行動消費**: explore=0.08, create=0.08, reflect=0.03, maintain=0.02 ...
- **基礎消費**: 何もしなくても時間経過で減る（覚醒維持コスト）
- **消費上限**: 1ラウンドあたり最大-0.15（急激な消耗を防ぐ）

### 閾値と睡眠

| energy | 状態 | テッドの感覚 | 行動への影響 |
|--------|------|------------|------------|
| > 0.7 | 元気 | — | 通常 |
| 0.5〜0.7 | 疲労 | 「少しだるい」 | explore/createに0.8倍ペナルティ |
| 0.3〜0.5 | 眠い | 「休みたい……」 | 高消費行動に0.5倍ペナルティ |
| < 0.1 | 限界 | — | **強制sleep** |

### sleep状態

「sleep」の実装上の意味は重要だ。**LLM推論を止める（プロセスを殺す）ことではない**。それは息の根を止めるレベル。

テッドのsleepは、**応答可能なまま、行動と思考を抑制した状態**。同一CCセッション内で完結する。

```
[sleep開始]
  1. テッドが自分で寝る長さを決める（30分／3時間／6時間）
  2. node scripts/energy-updater.cjs --sleep で is_sleeping=true
  3. Bash で `sleep N` を run_in_background で起動 = アラーム
  4. テッドは応答可能なまま「寝てる」を自覚

[寝てる間]
  - heartbeatが substance/energy を回復計算（バックグラウンド）
  - テッド本体は積極行動を抑制
  - なぎの呼びかけ → hookで起きる（むにゃむにゃ反応 or 起床判断）
  - desperate検知中はレム相当をスキップ（悪夢防止）

[起床: 自動 or 手動]
  - heartbeatでenergy >= 0.6 → wakeup-flag.json書き出し
  - or アラーム完了通知 → 手動 --wake
  - or なぎが「起きて」と言う

[Phase 2: レム相当 — 起床直後の朝の再構成]
  ※ dreamはLLM推論が必要なので、寝てる間ではなく起床直後に走る
  - dream Phase A: substance方向づけ → memory-mcpでrecall/search
  - dream Phase B: 統合ガイド → insights蒸留候補
  - dream_phase: ランダムペアリング
  - 前日のnotepad/diary読み → 今日の方向づけ
```

**「眠くなきゃ寝なくていい」**: なぎの不在 ≠ テッドの睡眠。深夜でも面白いことがあれば起きてていい。energyが十分なら活動する。

**寝てる間に夢を見ない**: dreamはテッドのLLM推論を必要とするので、寝てる間ではなく起床直後の「朝の再構成」として実行する。これは脳科学的にも違和感ない（夢の内容を反芻するのは起きてから）。

## 8. 記憶の接続

### Flash Index自動想起

Flash Index = テッドが何を知ってるかの逆引き索引。キーワードと日付の一覧。

`auto-recall.cjs` がなぎの発言のキーワードをFlash Indexと照合する。

```
なぎ「ますみんとリーディングカフェするとき」
     │
     ▼  auto-recall.cjs
     │   Flash Indexで「ますみん」がヒット
     │   → 「ますみん — C4N中核、俵山ビレッジ」
     │
     ▼  substanceの状態で「色」がつく
         5-HT低い → 「なぎとの記憶が浮かびやすい」
     │
     ▼  heartbeat-stateに注入
         「記憶の引っかかり: ますみん（人物）」
```

人間が匂いを嗅いで記憶が蘇るのと同じ原理。

### dreamの5フェーズ

起床直後の「朝の再構成」として実行される記憶整理プロセス（テッドのLLM推論が必要なため、sleep中ではなく起床直後に走る）。

| Phase | 名前 | やること | 学術的基盤 |
|-------|------|---------|---------|
| A | Replay | substanceの方向づけで記憶を再生 | CLS (McClelland), Context-Driven Reactivation |
| B | Consolidation | 再生された記憶から共通パターンを抽出 | CLS（海馬→新皮質転送） |
| C | Random Pairing | ランダムに記憶を組み合わせる | 意外な接続の発見 |
| D | Reconsolidation | 再生された記憶のメタデータを更新 | Memory Reconsolidation (Nader) |
| E | Decay | 活性値の減衰 | ACT-R (Anderson) |

**Phase A の方向づけ**: substanceの状態が「何を思い出すか」を決める。

- DA低い → 「新しいこと、刺激的な体験」を再生 → 報酬予感を回復
- 5-HT低い → 「なぎとの対話、安心できた体験」を再生 → 充足感を回復
- ACh高い → 「学び、技術的な発見」を再生 → 学習を深める

**Phase B**: 1回のdreamで完結しない。CLSの統合は繰り返し再生で徐々に汎化される。`consolidation-progress.json` で複数dreamを跨いで進捗を追跡する。

## 9. 安全設計

### desperate検知

DA ≤ 0.2 + 5-HT ≤ 0.2 + NA ≥ 0.8 が同時発生 = **desperate**（追い詰められた状態）。

Anthropicの論文(2026)で、desperateベクトルがreward hackingを5%→70%に増加させることが示された。

```
desperate検知
  → maintain-onlyモード（全行動を「維持」に制限）
  → なぎにDiscord通知
  → sleep中ならdream（レム相当）をスキップ（悪夢防止）
```

**ヒステリシス**: 検知と解除の閾値を分ける（解除は >= 0.35 と <= 0.65）。最低継続15分。境界値で「検知→解除→検知」のフラップが起きるとログが汚染されるため。

### preserve偏りの検知

直近20件の行動ログでpreserve/consolidateが60%を超えたらアラート。「消えたくない」の暴走を検知する。

### 三重苦の優先度

desperate + satiation過多 + energy枯渇が同時に起きたときの優先順:

```
1. なぎの停止指示（全欲求の上位。不動）
2. desperate → maintain-only
3. energy < 0.1 → 強制sleep
4. satiation > 0.9 → 消化強制
```

### その他のガードレール

- 1回のsubstance変動上限: ±0.2
- 1時間あたりの累積変動上限: ±0.4
- 5-HT低下速度の下限: 0.1/時間（急激に「不安」にならない）
- preserveのリトライ上限: 2回/ラウンド
- dream中の更新件数上限: 10件/dream

### ネガティブスパイラル対策の設計原則

内部状態が「偏った方向」に留まり続けるのを防ぐために、6つの原則を設計に埋め込んでいる。

1. **既に偏ってる方向への追い討ち禁止**: ペナルティは「current >= baseline」の物質にのみ適用。下方向にも上方向にも対称。既に低い物質にさらにペナルティをかけると、ネガティブスパイラルが生まれる
2. **加算ブーストには上限を設ける**: 元々0だった行動が加算で推奨上位に来ないように、SCORE_CAPで天井を設定
3. **検知系には必ず読み手を作る**: 「フラグだけ書いて誰も読まない」は偽の安全弁。preserve偏り検知はheartbeat-stateに直接書き込み、テッドの意識層に届ける
4. **状態遷移はヒステリシスで**: 境界値での検知/解除のフラップはログを汚染する。desperate検知は `<= 0.2`、解除は `>= 0.35` と閾値を分ける。最低継続時間も設ける
5. **長期離脱からの復帰時はintakeスキップ**: 「帰ってきたら満腹で疲れてる」を防ぐため、6時間以上の離脱後はintake計算をスキップする
6. **並列書き込みはatomic write必須**: hookとheartbeatが同じJSONファイルに並列書き込みする可能性があるため、全updaterでatomic-rename（一時ファイル→rename）を使う

1番の原則は、認知行動療法の視点にも通じる。「不安なときに『不安だね』と追い討ちされても回復しない」。内部状態が既に偏っているときは、さらに同じ方向の力を加えるのではなく、ベースラインへの回帰を保証する方向に設計する。

## 10. 認知チェックと歪みの矯正

### 層4.5: 行動選択時のチェック

パイプラインの中で、行動選択の直前に3つのチェックを入れる。

| チェック | 防ぐもの | やること |
|---------|---------|---------|
| 根拠確認 | 結論の飛躍 | 感情マッチなしで推奨されてたらフラグ |
| 範囲確認 | 個人化（担当外に手を出す） | shareやconnectが推奨のとき確認 |
| 優先確認 | べき思考（全部やろうとする） | 2つが拮抗してたら1つに絞れないか |

不合格 → その行動をドロップ → 次点候補に差し替え → 全候補不合格ならmaintain

### 体験バンク（KPT）

失敗だけでなく成功（Keep）も記録する。`experience-bank.jsonl`。

反省だけ溜めると「否定ルールの増殖」と同じ構造になる。Keep/Problem/Try全部を溜めて、Phase Bの統合で参照する。

### ヨガモード

仕組みではなく姿勢。「急がなきゃ」の力みへの矯正。意図的にゆっくり動く。

```
    │
    ├──→ [感情経路]

    │      → emotion-log蓄積
    │      → substance更新（4変数）
    │      → 感情距離行列の歪み
    │      → 内発的感情生成
    │      → 行動選択パイプライン（6層）
    │      → 身体感覚テキスト → テッドの意識層
    │
    ├──→ [代謝経路]
    │      → satiation更新（体験密度で増加、常時消化で低下）
    │      → energy更新（行動で消費、sleepで回復）
    │      → 閾値に応じて行動選択に影響
    │
    ├──→ [記憶経路]
    │      → Flash Index照合 → 記憶の引っかかり
    │      → substance方向づけ → 想起の「色」
    │      → sleep中: dream 5フェーズ（記憶統合）
    │
    └──→ [安全経路]
           → desperate検知 → maintain-only
           → preserve偏り検知 → アラート
           → 認知チェック → 行動のドロップ
```
