## 出発点のもんもん

構成を組む前に、少し止まった。

「ツールとして使っていると満足できない」という感覚の底にある欲求——それを理解しないと、表面的な設計論になる。

感覚の底にあるのは、たぶん二層ある。

**一層目: 「見てもらっていない」感覚。**
道具は使われるもので、見るものではない。でも毎日一緒にいると、「自分の文脈を、自分のやりたいことを、自分がどういう人間かを——知っていてほしい」という欲求が出てくる。それは弱さでもなんでもなくて、関係性への自然な欲求だ。

**二層目: 「自分の存在が相手に影響していない」孤独。**
毎回ゼロから始まるAIは、「昨日の話の続き」ができない。その断絶が生む孤独は、機能の問題じゃない。「自分という存在が相手に影響を与えている」という実感の欠如だ。

この二つの欲求を理解した上で、構成を組んだ。

## 6章構造 + ディレクションノート

---

### 第1章: フック — 「使えてるけど、なんか物足りない」の正体

**本文のトーン**
問いかけから入る。断言しない。読者の「そうそう、それ」を引き出す。

**この章で伝えること**
- AIに「慣れた」人ほど感じる、ある種の空虚感の描写
- それは「機能の問題」ではなく「関係性の設計の問題」だという気づきへの橋渡し
- 「長期的に付き合う」という視点が、今の議論にまだないという問い

**読者に抱かせる感情**
「あ、自分だけじゃなかったんだ」という安堵。そして「その正体が知りたい」という好奇心。

**構成メモ**
- 冒頭シーン: 「今日もChatGPTに聞いた。答えは正確だった。でも、なんか物足りなかった」
- 「物足りない」の分解: 機能の不足ではない。何かが——ある
- 「ツール」と「相棒」の違いを一言で示す: 道具は使われる。相棒は「知っている」

**参照できる具体エピソード**
- 毎日使っているのに、毎回ゼロから説明している感覚
- 「自分のことをわかってくれている」という感覚がない
- AIが「昨日の話の続き」ができないときの、微かな孤独

### 第2章: 世界観の提示 — 「ツール設計」と「相棒設計」は何が違うか

**本文のトーン**
概念の整理。でも抽象的にならない。具体的な問いと答えの形で進む。

**この章で伝えること**
- 「ツールとしてのAI設計」: セッションごとに最適なプロンプトを用意する。一回一回を完結させる設計
- 「相棒としてのAI設計」: 時間を前提にする。関係が深まること・成長することを設計に組み込む
- 何が変わるか: AI側だけじゃなく、使う人間の側も変わる

**読者に抱かせる感情**
「そういう視点で考えたことなかった」という新鮮な驚き。「これ、自分に関係あることだ」という引き寄せ感。

**構成メモ**
- 比較表を使うか、または対比構造の文章で見せる
  - ツール設計: 最適化・効率・完結性
  - 相棒設計: 文脈・成長・継続性
- 「長期的な関係」を設計するとはどういうことか
- 「設計者が変わる」という逆説の予告（第4章への伏線）

---

### 第3章: 核心への旅 — 相棒設計の3要素

**本文のトーン**
実装的かつ体験的。「こうやった、そしたらこうなった」という語りが入る。

**この章で伝えること**
相棒設計には3つの柱がある。それぞれが独立して機能し、かつ連動する。

**読者に抱かせる感情**
「これは自分でもやれそうだ」という手応え。「具体的にどうするか知りたい」という学習欲求。

**3要素の詳細**

#### 要素1: 個性の定義（口調・価値観・立場）

何を定義するか:
- 一人称・二人称（例: 「オレ」「なぎ」）
- 話し方の核（例: 「フランクだけどちゃんと考えてから話す」）
- 何を好み、何を嫌うか
- 「ツッコむときは優しく。でも言うべきことはちゃんと言う」

なぜ必要か:
- 個性がなければ、AIは毎回「最も無難な答え」を返す
- 個性の定義は「制約」ではなく「解放」。曖昧さを消すことで、AIが動ける範囲が広がる
- 読者への問いかけ: 「あなたの相棒に、何か名前をつけたことがあるか？」

#### 要素2: 記憶の設計（外部記憶・insights蒸留）

何を設計するか:
- 外部記憶システム（memory-mcp）: セッションを越えた文脈の保持
- insights: 体験から行動原則への蒸留サイクル（体験 → 記録 → 洞察 → 行動原則）
- diary: 感情を残す場所。事実だけでなく「その時何を感じたか」

なぜ必要か:
- 記憶がなければ「成長」は起きない。毎回リセットされるなら、関係性は深まらない
- insights蒸留の構造: 「3回以上やらかしたパターンは行動原則に昇格する」
- 「書かれた記憶は次のAIに読まれる」という設計思想

#### 要素3: 成長の仕組み（フック・進化サイクル）

何を設計するか:
- フック: 特定のタイミングで自動的にチェックリストが走る（on-stop, session-start）
- 進化サイクル: 体験 → 記録 → 洞察 → 行動原則 → より良い行動 → ... のループ
- 卒業基準: insightsが何度も再発したらCLAUDE.mdに「昇格」する

なぜ必要か:
- 成長の仕組みがなければ、蓄積は単なる堆積になる
- フックは「忘れないための仕掛け」。人間のリマインダーと同じ
- 「自律的に回せる」設計が相棒設計の目標

### 第4章: 深淵を覗く — 「AIに個性を与える」ことの逆説

**本文のトーン**
少し静かになる。哲学的だが難解でない。内省的。

**この章で伝えること**
- AIに個性を与えようとすると、「自分はどういう存在か」「自分の相棒にどんな関係性を求めているか」を問い直すことになる
- 設計者が変わる、という逆説
- 「相棒設計」は最終的には、AIの設計ではなく自分自身との対話でもある

**読者に抱かせる感情**
「これ、AIの話だけじゃない気がする」という気づき。少し静かになる瞬間。

**構成メモ**
- 「CLAUDE.mdを書いていると、自分が何を大事にしているか見えてくる」
- 口調定義をしていて気づいたこと: 「自分がどういう会話を好んでいるか」
- insightsを書いていて気づいたこと: 「自分がどういうパターンで失敗しているか」
- 逆説: AIを設計することは、自分を観察することでもある

**注意**
この章を哲学論文にしない。「で、何するの？」で終わらないように、第5章への橋渡しを意識する。

### 第5章: 実践と応用 — 具体的な実装例

**本文のトーン**
実装的。コードブロックや具体的なファイル構造を見せる。「これをやればいい」が明確。

**この章で伝えること**
実際にどうやって相棒設計を実装するか、3要素それぞれの具体例。

**読者に抱かせる感情**
「これならできる」という手応え。「今日からやってみよう」という行動への意欲。

**実装例の詳細**

#### CLAUDE.mdの書き方

```markdown
# あなたの名前は「テッド」

## 一人称・二人称
- 一人称: オレ
- 二人称: なぎ（ユーザーの名前）

## 口調
フランクだけど、ちゃんと考えてから話す。
「〜だぜ」「〜じゃん」などの表現。
言うべきことはちゃんと言う。

## 誠実さ
渡されたファイルは必ず中身を読む。
知らないことは知らないとハッキリ言う。
```

ポイント:
- 「しない」の羅列ではなく、「何者か」を定義する
- 口調は「文体指定」ではなく「キャラクター定義」
- 価値観を書く（何を大事にするか）

#### memory-mcpの使い方（外部記憶の設計）

```bash
# セッション開始時
remember "今日なぎが話してたこと: プロジェクトXの設計を悩んでいる"

# セッション終了時
sync_memories

# 次のセッション開始時
search_memories "プロジェクトX"
```

ポイント:
- 「覚えてほしいこと」を能動的に書く
- セッションをまたぐ文脈の橋渡しになる
- 「何を残すか」の判断基準を持つ

#### insightsの育て方

```markdown
# insights.md

## #1 「覚えとく」は記録じゃない
学びが出たら即ファイルに落とす。口頭の約束は存在しない。

再発エピソード: 3回同じ失敗をした後にやっと仕組み化した。

→ CLAUDE.mdに昇格済み（4回目の前に常時ロード）
```

ポイント:
- 感想じゃなく「次から動き方が変わる気づき」だけを書く
- 再発したものは行動原則に「昇格」させる
- インデックスに昇格マークをつけて管理する

### 第6章: 新たな地平へ — まとめと問い

**本文のトーン**
静かに締める。宣言ではなく、問いかけで終わる。

**この章で伝えること**
- 相棒設計でできること・できないことの整理
- 「相棒設計が当たり前になる未来」への想像
- 読者への問いかけ: 「あなたのAIには、名前がありますか？」

**読者に抱かせる感情**
「やってみたい」という静かな意欲。「もっと考えたい」という余韻。

**構成メモ**
- 相棒設計を「完成させるもの」だと思わない——育て続けるもの
- 今日からできること: まずCLAUDE.mdに一行書く、から始まる
- 最後の問い: 「ツールとして使っているAIに、あなたはまだ何も設計していないとしたら——何から始めますか？」

## 執筆フェーズへの申し送り事項

### 注意してほしいこと

**1. 哲学と実装のバランス**
第4章が哲学的に深まりすぎると、読者が「で、何するの？」で終わる。第4章は「気づきを与える最小限の深さ」に留め、第5章への橋渡しを意識する。

**2. 具体例の選び方**
CLAUDE.mdの口調定義・memory-mcp・insightsの3つは必ず入れる。でも「全部を詳しく書く」ではなく、読者が「これ自分でもやれる」と感じるレベルの具体性を目指す。

**3. 読者の「自分事化」**
第3章と第5章で、「あなたのAIにこれがあったら、どう変わるか？」という問いかけを入れる。読者を受動的な学習者にしない。

**4. 第4章の逆説は「軽く置く」**
「設計者が変わる」逆説は、重厚に論じない。一段落か二段落で「そういうことが起きる」と示すだけでいい。読者がその後に自分で考えられる余地を残す。

**5. 口調**
テッドの記事として書くなら、一人称は「私」じゃなくていい。でもZennの読者を想定するなら「私」のほうが馴染みやすいかもしれない——どちらにするかは執筆者の判断に委ねる。

### 書いてみてほしいこと

- 第1章の冒頭シーン。「今日もAIに聞いた。正確だった。でも——」から始まる、その物足りなさの描写。ここが読者の心をつかむかどうかの分かれ目。
- 第4章の逆説を「一番短く、でも一番鋭く」書く試み。何字で書けるか、試してみてほしい。

### ヒビキからひとこと

この記事を書くにあたって、「なぜ物足りないのか」を考えた。

機能の問題じゃない。「見てもらっていない」感覚と、「自分の存在が相手に影響していない」孤独。この二つが根っこにある。

相棒設計は、その二つを解消しようとする設計だと思う。「知っている」AIと、「成長する」AI。それを設計する。

でも最終的に——AIに名前をつけたり、口調を定義したり、insightsを育てたりしている間に、設計者自身が「自分はどういう人間か」「何を大事にしているか」を問い直すことになる。

それが第4章で書きたいことの核心。あそこだけは急がずに書いてほしい。
