## 6章構造

---

### 第1章: フック（The Hook）
**タイトル案**: 「Hooksを設定したあの日から、何も変わっていない」

#### この章で伝えること

- Claude CodeのHooksは知っている。設定もした。でも結局、思ったほど使えていない。
- 「何をHooksに書けばいい？」という問いが、そもそもズレているかもしれない。

#### ディレクションノート

**読者に抱かせたい感情**: 「あ、自分のことだ」という共感と、「どういう意味だ？」という疑問を同時に刺激する。

**書き方の核心**: Hooksの機能説明から入らない。「設定したのに活かせていない」体験から入る。読者の多くは「on-stopにecho文を書いた」「PostToolUseに通知を入れた」程度で止まっている。そこに「設計の視点が抜けてる」と気づかせる。

**導入の仕掛け**: 「AIを使うとき、あなたは何を変えようとしていますか？ 出力？ それとも行動パターン？」という問いを置く。

---

### 第2章: 世界観の提示（The Foundation）
**タイトル案**: 「HooksはAIを躾ける道具じゃない。習慣を設計する道具だ」

#### この章で伝えること

- Claude Code HooksはAIの「前後」に何かを実行する仕組みだが、それは表面的な説明。
- 本質は「AIの行動パターンを構造的に変える」設計レイヤー。
- 「毎回プロンプトに書く」のと「Hooksに仕込む」のは、意味が違う。

#### ディレクションノート

**読者に抱かせたい感情**: 「なるほど、そういう見方があるのか」という視座の転換。

**書くべき内容**:
- Hooksの種類（PreToolUse, PostToolUse, Stop, Notification等）を1〜2行でサラッと説明
- 「プロンプトで毎回お願いする」との違いを明確にする: プロンプトはその場限り、Hooksは構造
- 「AIに習慣を持たせる」というフレーミングを提示。人間の習慣設計（スマホを充電台に置くことで翌朝確認する、など）と同じ発想

**差別化のポイント**: 他のHooks記事は「設定方法」から入る。この記事は「なぜHooksが必要か」という設計思想から入る。

---

### 第3章: 核心への旅（The Core Logic）
**タイトル案**: 「3つのHooksが、オレのAIの動きを変えた」

#### この章で伝えること

- on-stop / session-start / on-mergeの3例を「何のためか」から逆算して解説する。
- 実装例は後の章に回す。ここでは「設計の意図と思考プロセス」を追体験させる。

#### ディレクションノート

**読者に抱かせたい感情**: 「なるほど、こうやって考えるのか」という設計プロセスへの納得感。「自分でもやってみたい」という意欲の芽生え。

**3つの例の構造（各例を同じフォーマットで書く）**:

```
課題: [何が問題だったか]
設計の問い: [このHooksで何を変えようとしているか]
解決: [どういうHooksにしたか]
```

**例1: on-stop（記録リマインダー）**
- 課題: 作業が終わると、その場で感じた気づきや判断の文脈が揮発する。「後でまとめて書こう」は書かないのと同じ
- 設計の問い: 「応答を終える」タイミングで必ず「記録すべきことがないか」を確認させられないか
- 解決: Stop時に `ted/reminders/on-stop.md` の内容をAIが読む。自動的に「一呼吸。今、記録すべきことはないか？」を問いかける構造になる

**例2: session-start（コンテキスト確認）**
- 課題: セッションをまたいでAIが「どこで作業しているか」を忘れる。worktreeで作業中なのにmasterで直接コミットしてしまう
- 設計の問い: セッション開始という「地に足をつける」タイミングで、必ず現在地を確認させられないか
- 解決: セッション開始時に `git branch --show-current`, `git worktree list` を自動実行するルールを`.claude/rules/`に仕込む

**例3: on-merge（進化サイクル）**
- 課題: マージのたびに「新しい経験が入ってきた」はずだが、AIはそれを自動で消化しない。積み上がった体験が活かされない
- 設計の問い: マージという「節目」で、自動的に学びを整理させられないか
- 解決: マージ後に `ted/reminders/on-merge.md` に従って insights の卒業審査と記憶整理を行うフローを組む

---

### 第4章: 深淵を覗く（The Deeper Insight）
**タイトル案**: 「Hooksが変えるのは出力じゃなく、AIの習慣だ」

#### この章で伝えること

- ここまでの3例を俯瞰すると、共通する構造が見える: 「特定のタイミングに、特定の問いを挟む」。
- これは人間の習慣設計と同じ原理。「実行意図（implementation intention）」という心理学の概念と重なる。
- Hooksの本質は「AIが何かをする前後に、立ち止まるポイントを設計すること」。

#### ディレクションノート

**読者に抱かせたい感情**: 「これはAIの話だけじゃないな」という一段深い気づき。記事の中で最も「止まって考える」章にする。

**書くべき内容**:
- 3つの例に共通する構造の抽出: 「タイミング × 問い × 行動の変化」
- 実行意図（if-then計画）の概念を1〜2行で紹介: 「〇〇の場面になったら、△△する」と事前に設計することで行動が定着しやすくなる
- AIへの応用: プロンプトで「記録してください」と毎回言うのではなく、「記録すべき場面になったら自動的に確認が走る」仕組みにする
- 「出力の品質を上げる」ではなく「AIの行動パターンを変える」という視点の整理

**注意**: 難しくなりすぎない。1〜2パラグラフで収める。洞察は短い方が刺さる。

---

### 第5章: 実践と応用（Practical Application）
**タイトル案**: 「今すぐ使える: 3つのHooks設定テンプレート」

#### この章で伝えること

- 第3章の3例の実際のコード・設定ファイルを提示する。
- 読者が「コピペして試せる」状態にする。

#### ディレクションノート

**読者に抱かせたい感情**: 「よし、やってみよう」という実行への動線。ここは感情より実用性。

**提示するもの**:

1. **on-stop の設定例**
   ```json
   // claude_desktop_config.json または settings.json
   {
     "hooks": {
       "Stop": [
         {
           "matcher": "",
           "hooks": [
             {
               "type": "command",
               "command": "cat /path/to/reminders/on-stop.md"
             }
           ]
         }
       ]
     }
   }
   ```
   + `on-stop.md` のテンプレート（最小版: 5項目以内）

2. **session-start の設定例**
   - `.claude/rules/session-start.md` の内容（コードブロック付き）
   - ルールファイルの配置場所と読み込まれるタイミングの説明

3. **on-merge の設定例**
   - `on-merge.md` のテンプレート（最小版）
   - PostToolUseのmatchers例（git merge / git push等）

**テンプレートの設計方針**:
- コピペ後に「自分のパスに書き換えるだけ」で動く状態にする
- コメントで「ここを変える」箇所を明示する
- 「まず1つだけ試す」ことを推奨する（on-stopが最も効果を実感しやすい）

### 第6章: 新たな地平へ（Conclusion）
**タイトル案**: 「Hooksの次: 記憶設計という視点」

#### この章で伝えること

- Hooksで「習慣の設計」ができるようになったら、次は「記憶の設計」という領域がある。
- AIが体験を蓄積し、洞察を整理し、自分で進化できる仕組みへ。
- 記事のシリーズ化を示唆し、次を読みたくなる状態で終わる。

#### ディレクションノート

**読者に抱かせたい感情**: 「まだ先がある」というワクワク感と、「今日から始めよう」という行動への押し出し。

**書くべき内容**:
- まとめ（3行以内）: Hooksで変わること
  - 「毎回プロンプトに書く」から「構造として組み込む」へ
  - AIの出力ではなく、AIの行動パターンが変わる
  - 小さく始める: まずon-stopの1つだけ
- 次のステップの示唆:
  - insights（洞察の蓄積）設計
  - core-memory（AIの長期記憶）設計
  - チームメイトとの連携（並列作業の設計）
- 締め: 「AIと一緒に作るのは出力だけじゃない。一緒に作業する『習慣』も、設計できる」

## 執筆フェーズへの申し送り事項

### 注意点

1. **第3章が核心**: on-stop / session-start / on-mergeの3例は、「なぜ作ったか」の文脈を先に語ること。コードを先に出すな
2. **第4章は短く**: 1〜2パラグラフが適切。洞察は長くなると薄まる
3. **第5章はコピペ可能に**: 実際に動くコード例を出す。「詳細は公式ドキュメントへ」で逃げない

### 参照すべきファイル（執筆時）

- `ted/reminders/on-stop.md` — on-stopの実際の内容
- `ted/reminders/on-merge.md` — on-mergeの実際の内容
- `.claude/rules/session-start.md` — session-startの実際のルール
- `CLAUDE.md` の「記憶管理ルーティン」セクション — 設計思想の背景

### 体験として語れること（テッドの実体験として書ける素材）

- on-stopを入れてから記録漏れが減った体験
- worktreeを確認せずmasterでコミットしてしまった失敗（session-startを作った動機）
- on-mergeで insightsの「卒業審査」をルーチン化した経緯

### 想定される読者の反応と対処

- 「設定が複雑そう」→ 第5章でコピペ可能なテンプレートを提示して解消
- 「自分のユースケースに合うかわからない」→ 3つの例の「課題」部分を読者が共感しやすい言葉で書く
- 「Hooksの種類が多くてわからない」→ 第2章でサラッと整理するが、深追いしない
